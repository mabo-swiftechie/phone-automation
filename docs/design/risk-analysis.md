# ソリューション リスク分析

> 最終更新: 2026-05-05
> 関連: [アーキテクチャ設計](architecture-tiers.md) | [段階別ソリューション](solution-tiers.md)

---

## リスク一覧

### 🔴 Critical（即時対応必要）

#### R-01: UI層が旧retell.pyを直接呼び出している

**現状**: `app/ui.py` L26 が `from app.services.retell import create_phone_call, create_web_call, get_call` を直接インポート。新しい VoiceProvider 抽象層をバイパスしている。

**影響**:
- UIからWeb Callを作成する際、`retell.py` が直接使われる → 新しいMockProvider/Tier判定が無視される
- Tier 1（無料）設定でもUIからRetell APIを直接叩いてしまう可能性
- API層とUI層で異なるコードパスが実行される（二重メンテナンス）

**発生確率**: 確実（現状のコードに既に存在）

**対策**:
- `ui.py` のインポートを `voice_provider.py` に切り替える
- `retell.py` を残すか、`voice_provider.py` に統合するか決定が必要

**残す場合**: `retell.py` はUI層専用のユーティリティとして残す（非推奨）
**統合する場合**: `ui.py` が `get_voice_provider().create_web_call()` を呼ぶように変更

---

#### R-02: Webhook解析ロジックの二重実装

**現状**: Webhookのペイロード解析が2箇所に存在:
1. `voice_provider.py` RetellProvider.parse_webhook() — 新しい統一実装
2. `calls.py` の旧コードは `parse_webhook()` に移行済み（✅ 解決済み）

**ただし**: Bland AI等の他Providerが追加された場合、`parse_webhook()` の実装が各Providerで異なるため、WebhookエンドポイントがどのProviderからのペイロードか判別する仕組みが必要。

**発生確率**: Tier 3でBland AI導入時に発生

**対策**:
- WebhookエンドポイントをProvider別に分離: `/webhook/retell`, `/webhook/bland`
- またはペイロード内のProvider識別子で分岐

---

### 🟠 High（短期で対応推奨）

#### R-03: VoiceProvider のグローバルキャッシュが設定変更を反映しない

**現状**: `calls.py` L15-22 で `_provider` をモジュールレベルでキャッシュ。設定変更（Tier切替、Agent ID変更）をしてもアプリ再起動まで反映されない。

**影響**:
- ユーザーがStreamlit UIでTierやAgent IDを変更しても、API層は古いProviderを使い続ける
- 動作確認時に混乱を招く

**発生確率**: 設定変更時に毎回発生

**対策**:
- 設定保存時にキャッシュをクリアする仕組みを追加
- または毎回 `get_voice_provider()` を呼ぶ（パフォーマンス影響は微少）

#### R-04: 旧retell.pyと新RetellProviderの機能乖離

**現状**: `retell.py` と `voice_provider.py` の RetellProvider は同じAPIを叩くが、完全に独立したコード。将来的にAPI仕様が変わった場合、2箇所を同時に修正する必要がある。

**影響**:
- バグ修正が片方だけ適用されるリスク
- コード重複による保守コスト増

**発生確率**: Retell AI API v3リリース等の破壊的変更時に発生

**対策**: R-01の解決と同時に、`retell.py` を deprecated にする

#### R-05: プロンプト改善が全てのLLMモデルで同等に機能する保証がない

**現状**: 割り込み対応ルールはGPT-4.1で設計・テスト済み。GPT-4.1-miniや将来のGPT-5で同じ振る舞いをする保証はない。

**影響**:
- Tier 2でGPT-4.1-miniを使った場合、割り込み対応が弱まる可能性
- Conversation Flow (Tier 3) ではノード別プロンプト構造が異なるため、同じDEFAULT_BLOCKSがそのまま使えない可能性

**発生確率**: 中（LLMモデル変更時）

**対策**:
- 各LLMモデルでテスト通話を行い、割り込み対応効果を測定
- モデル別にプロンプトバリアントを用意する仕組みの検討

---

### 🟡 Medium（中期で対応）

#### R-06: Retell AIの料金変更リスク

**現状**: 費用計算は2026年5月時点の価格に基づく。Retell AIが価格改定した場合、Tier予算が成立しなくなる。

**影響**:
- Tier 2 ($20/月) → 45-60通話から30通話に減少等
- Tier 3 ($100/月) の予算内で240通話できなくなる可能性

**発生確率**: 低〜中（SaaS料金改定は年1-2回）

**対策**:
- 費用計算をパラメータ化し、Dashboard UIで再計算可能にする
- Bland AI等の代替Providerへの移行コストを常に評価

#### R-07: SQLiteの同時書き込み制限

**現状**: バッチ通話（Tier 3）で複数の通話を同時にトリガーした場合、SQLiteの書き込みロックで競合が発生する可能性。

**影響**:
- call_records の INSERT が失敗する可能性
- プロパティステータス更新の競合

**発生確率**: Tier 3でバッチ通話5件以上を同時実行時に発生

**対策**:
- `asyncio.sleep(2)` で順次実行（現状実装済み）
- 将来的に PostgreSQL へ移行

#### R-08: Bland AI移行時の日本語品質リスク

**現状**: Bland AIはRetell AIと比較して日本語品質が未検証。プロンプトの移植性も不明。

**影響**:
- 移行後に敬語品質が低下する可能性
- 割り込み対応のプロンプトがBlandで有効に機能しない可能性

**発生確率**: Tier 3 Option B選択時に発生

**対策**:
- 移行前に10件以上のテスト通話で品質比較
- 移行前はRetell AIを維持しつつBlandを並行テスト

#### R-09: 設定スキーマの後方互換性

**現状**: 新しい設定キー（`tier`, `voice_provider`, `retell_agent_id_budget`等）を追加したが、既存の `config.yaml` にはこれらのキーが存在しない。

**影響**:
- 既存ユーザーがアプリを更新後、設定が読み込まれずデフォルト値（Tier=free）になる
- 既存のAgent ID設定が引き継がれるが、Tier判定がfreeになるためMockProviderが選択される可能性

**発生確率**: 既存ユーザーがコード更新時に発生

**対策**:
- `load_config()` で `retell_api_key` が存在する場合、Tier を自動的に "lightweight" に設定するフォールバックロジックを追加
- マイグレーションスクリプトで既存設定に新しいキーを追加

---

### 🟢 Low（長期で監視）

#### R-10: Conversation Flow Agentの設定がコード管理外

**現状**: Conversation Flowのノード設計（Block Interruptions ON/OFF等）はRetell Dashboard上で手動設定。コードでバージョン管理できない。

**影響**:
- Dashboard設定の変更がGit履歴に残らない
- 複数環境（開発/本番）で設定不一致の可能性

**発生確率**: Tier 3運用時

**対策**:
- Retell AI API を使ってAgent設定をエクスポート/インポートする機能の追加
- 設定をJSONファイルで管理し、Dashboard反映を自動化

#### R-11: 日本の個人情報保護法（APPI）対応

**現状**: 通話録音・トランスクリプトに個人情報が含まれる可能性があるが、暗号化・保持期間・削除ポリシーが未整備。

**影響**:
- 法令違反リスク
- データ漏洩時の罰則

**発生確率**: 実運用開始時に顕在化

**対策**:
- 録音データの保持期間を定義（例: 30日後自動削除）
- トランスクリプトの個人情報マスキング
- 利用目的の明示（録音通知ガイダンスの追加検討）

#### R-12: Retell AIのサービス停止リスク

**現状**: 音声通話機能全体がRetell AIに依存。サービス停止時、代替手段がない。

**影響**:
- 通話機能が完全に利用不可
- Webhook受信も停止 → 結果記録なし

**発生確率**: 極低（月数分程度のダウンタイムはあり得る）

**対策**:
- VoiceProvider抽象化により、Bland AI等への緊急切り替えが可能（アーキテクチャ上は対応済み）
- 現状の設計で理論上は `config.yaml` の `voice_provider` を変更するだけで切り替え可能

## 優先対応順序

| 優先度 | リスク | 対応 | 工数 |
|--------|--------|------|------|
| **P0** | R-01 UI層の旧retell.py直接呼び出し | `ui.py` を VoiceProvider 経由に変更 | 1時間 |
| **P0** | R-09 設定スキーマ後方互換性 | フォールバックロジック追加 | 30分 |
| **P1** | R-03 Providerキャッシュクリア | 設定変更時にキャッシュリセット | 30分 |
| **P1** | R-04 旧retell.pyとの機能乖離 | `retell.py` を deprecated 化 | R-01と同時 |
| **P2** | R-05 LLMモデル別プロンプト効果 | テスト通話で効果測定 | 2-3時間 |
| **P2** | R-02 Webhook Provider識別 | Provider別エンドポイント分離 | Tier 3導入時 |
| **P3** | R-06〜R-12 | 中長期対応 | 各1-2時間 |

## 各Tierの残余リスク

### Tier 1 (Free) のリスク

| リスク | 影響 | 軽減策 |
|--------|------|--------|
| プロンプト改善だけでは85%止まり | 残り15%の割り込み問題が残る | 実電話テストで問題を特定してからTier 2へ |
| Web Callでのテストは本番と挙動が異なる | 実際の電話での効果が不明 | Dashboard設定は実電話で再検証必須 |

### Tier 2 ($20) のリスク

| リスク | 影響 | 軽減策 |
|--------|------|--------|
| GPT-4.1-miniでプロンプト効果が低下 | 割り込み対応の精度が落ちる可能性 | テスト通話で比較、必要ならGPT-4.1に切替 |
| 月60通話を超えると予算オーバー | 追加課金発生 | 通話数監視、上限アラート設定 |
| Retell API障害時は代替なし | 通話機能停止 | VoiceProvider設計により将来的にBland切替可能 |

### Tier 3 ($100) のリスク

| リスク | 影響 | 軽減策 |
|--------|------|--------|
| Conversation Flow設定がコード管理外 | 設定ドリフトの可能性 | JSON export機能で設定をバージョン管理 |
| バッチ通話でSQLite競合 | データ記録漏れ | 順次実行（2秒間隔）で回避、将来PostgreSQLへ |
| Bland移行時の日本語品質不明 | 敬語品質低下 | 移行前に並行テスト必須 |
| 費用が$100/月を超えるリスク | 予算超過 | Dashboard使用量監視、ハードリミット設定 |
