# AI電話自動化 — アーキテクチャ設計文档

## 技術スタック選定

### 現在の構成

| 層 | 技術 | 理由 |
|---|------|------|
| UI | Streamlit | Python単一ファイルでブラウザUI、Mac/Win対応、個位数ユーザーに最適 |
| DB | SQLite | ローカルファイル、インストール不要、バックアップはファイルコピーのみ |
| メール送信 | Gmail SMTP | 無料、アプリパスワードで認証、追加コストなし |
| メール生成・解析 | OpenAI GPT-4.1-mini | 高品質な日本語敬語生成、返信構造化解析、コスト低 |
| 音声通話 | Retell AI | 日本語対応、Web Call無料テスト、API駆動、開発者フレンドリー |
| 設定管理 | YAML | 人間が読める、手動編集可能、Git管理から除外 |

### 選定プロセス（なぜこの構成か）

**前提条件：**
- ユーザー数：個位数
- プラットフォーム：Windows中心 + Mac
- 運用：一回設定して「開いて使う」
- コスト：ゼロコスト検証 → 必要に応じて段階的投資

**段階1：FastAPI + Supabase + ngrok（却下）**
- 理由：構成が複雑すぎる、ngrokの接続不安定、Supabaseの外部依存
- 個位数ユーザーにはオーバーエンジニアリング

**段階2：Streamlit + SQLite（採用）**
- ローカル完結、外部依存最小
- `./start.sh` 一発起動
- データは `data.db` ファイル1つ

**メール優先戦略：**
- メール問い合わせ = ゼロコスト（Gmail無料）
- 電話 = 有料だが、メール未回答物件のみに限定
- これにより「無料検証 → 有料本番」の段階移行が可能

## 音声AI プラットフォーム比較調査（2026-04）

### クラウド型（API利用）

| プラットフォーム | 日本語 | 料金/分 | 特徴 | 移行コスト |
|----------------|--------|---------|------|-----------|
| **Retell AI** (現在使用) | 良好 | ~$0.10-0.25 | API友好、Web Call無料テスト | — |
| **Bland AI** | 良好 | **$0.015** | 最安、API駆動 | 低 |
| **LiveKit Agents** | 優秀 | ~$0.05+ | オープンソース+クラウド、自建可能 | 中 |
| **PolyAI** | 優秀 | 企業料金 | 最も自然な会話 | 高 |
| **Google Dialogflow CX** | 優秀 | ~$0.004 | Googleエコシステム | 高 |
| **Amazon Lex + Connect** | 良好 | 変動 | AWS統合 | 高 |

### オープンソース型（自前構築）

| ソフトウェア | 種類 | 日本語品質 | ハードウェア要件 | 特徴 |
|-------------|------|-----------|----------------|------|
| **Pipecat** | ASR+LLM+TTS フレームワーク | 優秀 | CPU 4Core + 8GB | モジュラー設計、プロバイダー自由切替 |
| **LiveKit Agents** | 音声AIエージェント枠組み | 優秀 | 8Core + 16GB | 本番実績あり、電話統合対応 |
| **VoiceVox** | 日本語専用TTS | **最も自然** | GPU推奨 | 無料、日本語特化 |
| **Style-Bert-VITS2** | 日本語TTS | 非常に自然 | GPU | スタイル制御可能 |
| **VibeVoice** (Microsoft) | ASR+TTS | 良好 | GPU必須(7B) | 60分長時間ASR、研究目的 |

### 自前構築の構成例

```
Pipecat (フレームワーク)
├── ASR: Google Speech-to-Text または Whisper
├── LLM: OpenAI GPT-4.1-mini (現状流用)
├── TTS: VoiceVox (無料・最高品質日本語)
└── 電話: Twilio ($0.0746/分 → 日本)
```

**総コスト見込み**: ~$0.10-0.15/分 (主にTwilio電話代)

### 技術選択の意思決定フロー

```
通話量は？
├── 月数十件以下 → Retell AI で十分（現状維持）
├── 月数百件 → Bland AI に移行（1/7のコスト）
└── 月千件以上 → Pipecat + VoiceVox 自前構築を検討
    ├── GPUサーバーある？ → LiveKit Agents + VoiceVox
    └── ない？ → クラウドGPU（RunPod等）+ Pipecat
```

## 会話テンプレート設計

### 積木式アーキテクチャ

```
Block (積木) — 独立した会話ステップ
├── type: greeting / question / followup / closing / instruction
├── content: 実際の話術テキスト
└── 再利用可能、削除・編集自由

Template (テンプレート) — Block の有序集合
├── 積木を並び替えて組み合わせ
├── JSON インポート/エクスポート対応
└── 物件ごとにテンプレート割り当て可能

Prompt 生成 — Template → Block 内容を順番に結合
└── Retell AI の retell_llm_dynamic_variables に注入
```

### デフォルト7積木

| # | タイプ | 名前 | 内容 |
|---|--------|------|------|
| 1 | instruction | 敬語ルール | 敬語使用規範、挨拶方式 |
| 2 | question | 空室確認 | 空室の有無 + フォローアップ分岐 |
| 3 | question | 外国人確認 | 外国人入居可否 |
| 4 | question | 中国人確認 | 中国籍入居可否 |
| 5 | question | 入居条件確認 | 敷金/礼金/保証会社/賃料/入居日 |
| 6 | instruction | 対話ルール | 会話中の振る舞い（相槌・押し問答禁止等） |
| 7 | closing | 終了ルール | まとめ確認 → 感謝 → 終了 |

## ワークフロー

```
物件登録
├── 📧 メール確認ルート (無料)
│   AI邮件生成 → Gmail送信 → 返信受信 → AI解析 → 結果保存
│
└── 📞 電話確認ルート (有料)
    テンプレート選択 → Web Call開始 → AI通話 → 結果取得 → 結果保存
                                                  ↓
                                        結果一覧 → CSV ダウンロード
```

## 今後の拡張候補

1. **音声プラットフォーム移行** — Retell → Bland AI (コスト削減)
2. **自前構築** — Pipecat + VoiceVox (完全ローカル化)
3. **LINE通知** — 通話結果をLINEに自動通知
4. **バッチ処理** — 複数物件の一括メール送信
5. **スケジュール通話** — 定時自動発信
