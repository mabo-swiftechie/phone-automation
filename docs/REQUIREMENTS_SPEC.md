## 1. 核心目的 (Core Objective)

### 1.1 背景 (Background)

現状の不動産物件確認業務では、担当者が物件情報サイト（SUUMO、Homes、ATHOME等）から物件リンクを抽出し、管理会社・大家へ電話で空室状況・入居条件（外国籍可否、中国人入居可否等）を確認している。この手動プロセスは以下の課題を抱えている：

| 課題 | 影響度 |
|------|--------|
| 1件あたり平均5〜10分の通話時間 + 発信待ち | 労働コスト増大 |
| 営業時間（平日9:00〜18:00）への制約 | 確認件数の上限 |
| 日本語敬語（尊敬語・謙譲語・丁寧語）の適切な運用 | ヒューマンエラー・対応品質のバラツキ |
| 大家・管理会社側の「非通知」や「見知らぬ番号」への応答拒否 | 電話接通率の低下 |
| 確認結果の手動記録・Excel転記 | 転記ミス・情報漏れ |

### 1.2 目的 (Purpose)

**AI音声エージェント（AI Voice Agent）による自動発信システム**を構築し、手動電話確認業務を完全代替する。具体的には：

1. **物件リンク→自動発信→空室確認→結果出力** の一連のパイプラインを自動化
2. 日本語敬語を適切に运用した自然な会話を実現
3. 電話接通率（Pickup Rate）および情報取得成功率（Conversion Rate）を最大化
4. 確認結果を構造化データとしてExcel/LINE等へ自動出力

### 1.3 スコープ (Scope)

| In Scope | Out of Scope (Phase 1) |
|----------|----------------------|
| 自動発信・空室確認・条件確認 | 物件契約・申込手続き |
| 日本語敬語での自然な会話 | 不動産仲介業務全般 |
| 結果の構造化データ出力 | クレジットカード決済・契約書生成 |
| LINE通知・Excel出力 | 物件検索サイトへの自動投稿 |
| 録音保存・品質管理ダッシュボード | 法規制コンプライアンス自動判定 |

## 2. 業務プロセスモデル (Business Process Model)

### 2.1 現行プロセス (AS-IS)

```
[物件リンク収集] → [手動で管理会社電話番号を特定]
        ↓
[手動発信] → [敬語で空室確認・条件確認]
        ↓
[通話内容をメモ] → [Excelに手動転記]
        ↓
[結果を担当者に報告]
```

**平均処理時間**: 物件1件あたり約15〜20分（発信待ち含む）
**1日処理上限**: 1名あたり約20〜30件

### 2.2 目標プロセス (TO-BE)

```
[物件リンク入力] → [スクレイピング/手動入力で物件情報・連絡先を抽出]
        ↓                    ↓
[バッチ登録] ← ─ ─ [物件データベース（PostgreSQL/Supabase）]
        ↓
[AI Voice Agent が自動発信]
   ├─ 日本語敬語で開始「お世話になっております。〇〇物件について…」
   ├─ 空室の有無を確認
   ├─ 外国籍（中国籍）入居可否を確認
   ├─ 追加条件（敷金・礼金・保証人等）を確認
   └─ 会話終了
        ↓
[ASR転写] → [LLM意味解析] → [構造化データ抽出]
        ↓
[結果をDB保存] → [Excel自動生成 / LINE通知 / Web Dashboard表示]
```

**目標処理時間**: 物件1件あたり約2〜4分（通話時間）
**1日処理上限**: システム制限なし（並列発信可能）

#### 3.1.2 推奨アーキテクチャ戦略

**MVP推奨（改訂）**: **Retell AI** をPlatformとして採用し、以下の構成で構築：

```
┌─────────────────────────────────────────────────────────┐
│                  Retell AI (Orchestration)               │
│  ┌─────────┐  ┌──────────────┐  ┌───────────────────┐  │
│  │ ASR     │  │ LLM          │  │ TTS               │  │
│  │ (Deepgram│  │ (GPT-4.1    │  │ (OpenAI TTS or    │  │
│  │  Stream)│  │  recommended)│  │  Azure DragonHD)  │  │
│  └─────────┘  └──────────────┘  └───────────────────┘  │
│         │            │                    │              │
│         └────────────┼────────────────────┘              │
│                      │                                    │
│       Retell Platform: $0.055/min + Provider at-cost      │
│       Default Config Total: ~$0.13/min                    │
│       Concurrency: 20 free, +$8/month each               │
│       Batch Calls: +$0.005/dial                           │
└─────────────────────────────────────────────────────────┘
                       │
              ┌────────┴────────┐
              │   Twilio SIP    │
              │   (+81 03番号)   │
              │   $4.75/月      │
              └─────────────────┘
```

**選定理由**:
- **コスト最適**: Default構成で~$0.13/min。Vapi + BYOと同等以下のコストで管理画面付き
- **Conversation Flow Agent**: 構造化された会話フローを構築可能。不動産確認の定型会話に最適
- **Batch Calls**: 一括発信機能が標準搭載（+$0.005/dial）
- **Build/Test/Deploy/Monitor**: 開発→テスト→本番のワークフローが統合
- **SIP統合**: Twilio +81番号との連携が容易
- **Webhook**: リアルタイムイベントで結果を即座に取得可能
- **Simulation Testing**: 本番発信前に会話シミュレーションが可能

**代替案 — Vapi.ai（より柔軟な構成が必要な場合）**:
- BYOモデルで最高のカスタマイズ性
- Azure DragonHD + Deepgram + Claude等の任意組み合わせ
- Platform Fee $0.05/minでRetellより安価だが、管理機能が少ない

### 3.2 日本語特有の技術的課題

#### 3.2.1 TTS（Text-to-Speech）— 日本語敬語対応

| Provider | 日本語音声 | Neural品質 | 敬語適性 | 推奨 |
|----------|-----------|-----------|---------|------|

| OpenAI gpt-4o-mini-tts | カスタム指示で敬語レベル制御可能。marin, cedar推奨 | ★★★★★ | ★★★★★ | **推奨（敬語制御最強）** |
| Amazon Polly | Kazuha, Tomoko, Takumi (Neural); Mizuki (Std) — ※Generative/Long-Form非対応 | ★★★★☆ | ★★★☆☆ | 既存コードで利用済 |
| ElevenLabs | 日本語対応（複数Voice） | ★★★★★ | ★★★☆☆ | カスタムVoice作成可 |

**重要知見**:
- **Azure DragonHDLatestNeural**（Nanami/Masaru）は最新LLMベースの音声合成で最高品質。Nanamiは「customerservice」スタイルをサポートし、不動産電話に最適
- **OpenAI gpt-4o-mini-tts**は指示ベースの音声制御（「丁寧な接客トーンで」「敬語を使って」等）が可能。敬語制御において最も柔軟
- TTS自体の敬語対応だけでなく、**LLMが生成するテキストの敬語品質**に依存するため、System Prompt設計が最重要

#### 3.3.2 電話番号表示戦略と接通率への影響

> **これは本システムの成否を分ける最重要ポイントの一つである。**

日本では「非通知」や「見知らぬ番号」からの着信に対する応答率が極めて低い。以下の戦略を検討する必要がある：

| 戦略 | 説明 | 期待接通率向上 | コスト |
|------|------|-------------|--------|
| **固定電話番号表示** | 03/06等の固定局番を表示 | ★★★★★ | $4.75/月/番号 |
| **フリーダイヤル（0120）** | 着信課金番号を表示 | ★★★★★ | $25/月 + 着信課金 |
| **地域番号の使い分け** | 発信先と同エリアの局番を使用 | ★★★★☆ | 複数番号購入必要 |
| **発信者ID（CNAM）登録** | 事業者名を番号に紐付け | ★★★☆☆ | 追加手続き必要 |
| **SMS事前通知** | 発信前にSMSで「後ほど電話します」 | ★★★★★ | SMS送信料 |

**推奨**: 固定電話番号（03等）を表示 + SMS事前通知 の組み合わせ。

#### 3.3.3 AI発信に対する日本人の心理的抵抗

| 懸念 | 対策 |
|------|------|
| 「機械の声」と気づいて即切断 | 高品質Neural TTS + 適度な間（間合い）の実装 |
| 「個人情報を機械に話す」不安 | AIであることを開示しつつ、業務目的を明確化 |
| 「電話は人間がかけるもの」という文化的期待 | 冒頭「自動音声でお伺いしますが、〇〇について確認させていただきます」と丁寧に告知 |
| 敬語の崩れへの違和感 | LLM System Promptに敬語ルールを厳格に定義 |

**M3-01 System Prompt 設計ガイドライン**:

```
[Role] あなたは日本の不動産仲介会社の担当者として、管理会社・大家様に電話をかけて
       物件の空室状況を確認する役割を担います。

[敬語ルール]
- 常に丁寧語（〜です・ます・ございます）を使用
- 相手への配慮を示す表現を用いる（「お忙しいところ恐縮です」「お世話になっております」）
- 物件名は「〇〇物件」、相手は「〇〇様」と敬称をつける
- 否定形は直接的に言わず、柔らかい表現を使う
- 会話の冒頭は必ず「お世話になっております。〇〇（会社名）の〇〇と申します。
  お忙しいところ恐れ入ります。〇〇物件について少々お伺いしてもよろしいでしょうか。」

[確認項目]
1. 現在空室があるか
2. 外国籍（中国籍）の方の入居が可能か
3. 入居にあたっての特別な条件（敷金・礼金・保証会社等）

[制約]
- 一方的に話し続けず、相手の返答を待つ
- 相手が忙しそうな場合は「また後ほどお電話いたします」と丁寧に終了
- 物件の価格交渉や契約の話はしない
```
